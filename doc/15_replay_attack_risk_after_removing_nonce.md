# 去掉 nonce 后的重放攻击风险说明

## 文档日期

2026-04-21

## 背景

当前 Atlas 的网关签名和内部服务签名已从“`timestamp + nonce + signature`”模式，调整为仅保留“`timestamp + signature`”模式。

本次调整的直接收益是：

- 降低 Redis 在认证链路中的耦合度
- 简化网关与服务间签名协议
- 减少因为 nonce 状态校验带来的运行复杂度

但同步带来的安全变化是：

- 系统不再具备严格的“防重放（Replay Protection）”能力
- 系统当前具备的是“防伪造 + 防篡改 + 有限时间窗口约束”
- 系统当前**不具备**“同一合法请求在时间窗口内只能成功一次”的能力

---

## 当前签名机制的安全属性

### 仍然保留的能力

当前签名机制仍然可以提供以下保护：

1. **防伪造**
    - 攻击者如果不知道签名密钥，无法凭空构造一个合法签名请求。

2. **防篡改**
    - `method`、`path`、`payload`、`callerService`、`timestamp` 参与签名。
    - 请求一旦被篡改，签名校验就会失败。

3. **过期请求拦截**
    - 系统仍然对 `timestamp` 做时间窗口校验。
    - 超出允许偏移时间的旧请求会被拒绝。

### 已失去的能力

当前机制不再具备以下保护：

1. **防重复提交 / 防重放**
    - 只要请求仍处于合法时间窗口内，完全相同的请求可以被再次发送。
    - 服务端无法识别“这是刚才已经处理过的同一份请求”。

2. **一次性请求标识**
    - 没有 `nonce` 或唯一请求号后，签名只证明“这个请求是合法生成的”，不能证明“这个请求是第一次出现”。

---

## 什么是重放攻击

重放攻击是指：

- 攻击者截获一份原本合法的请求
- 不修改内容，直接再次发送
- 由于签名仍然正确、时间戳仍未过期，服务端会把它当成新的合法请求继续处理

这类攻击不依赖破解密钥，也不依赖篡改报文。

攻击者只需要满足两个条件：

- 拿到一份合法请求数据
- 在允许时间窗口内重新发送

---

## 在当前实现下，重放为什么会成立

当前签名校验基于以下信息：

### 网关请求

签名材料包含：

- HTTP Method
- Path
- Payload
- Timestamp

### 内部服务请求

签名材料包含：

- HTTP Method
- Path
- CallerService
- Timestamp

问题在于：

- 如果两次请求的这些字段完全一样
- 且两次请求都发生在允许时间窗口内
- 那么两次请求的签名值也完全一样

服务端无法知道：

- 这是“第一次请求”
- 还是“刚才那个请求又被发了一遍”

之前 `nonce` 的作用正是在这里：

- 每次请求都带一个唯一值
- 服务端记录该值是否已使用
- 第二次出现同一个 `nonce` 时直接拒绝

去掉 `nonce` 后，这个防线就不存在了。

---

## 典型攻击前提

在内网部署场景下，重放风险通常不是“互联网随便谁都能打”，而是满足以下前提后才成立。

### 前提 1：攻击者能够接触到合法请求

常见方式包括：

1. **内网抓包**
    - 某一段链路未启用 TLS
    - 代理层、旁路设备或调试环境可看到明文请求

2. **日志泄露**
    - 请求头、签名、payload、callerService、timestamp 被完整打印到日志
    - 调试日志、网关日志、Feign 日志记录了可直接复用的数据

3. **代理或网关缓存泄露**
    - 反向代理、中间件、APM、错误追踪系统记录了原始请求数据

4. **内部人员或被入侵节点读取请求**
    - 某个微服务实例、跳板机、容器、测试机被攻陷
    - 攻击者可以获取到已经发出的合法请求内容

### 前提 2：重放发生在允许时间窗口内

如果 `allowedSkewSeconds` 设置为 300 秒，那么攻击者拿到报文后，在 5 分钟内都可能重放成功。

时间窗口越大：

- 可用性越强
- 抗时钟漂移越宽松
- 但重放窗口也越大

---

## 对 Atlas 的实际影响分析

### 风险较低的场景

以下场景即使被重放，影响通常可控：

1. **纯读取接口**
    - 查询用户
    - 查询菜单
    - 查询字典
    - 查询配置

2. **天然幂等接口**
    - 多次执行结果相同
    - 重放不会造成额外状态变化

3. **只用于内部身份证明的轻量调用**
    - 主要目的是让下游相信“请求来自可信调用方”
    - 即使重复执行一次，业务后果较轻

### 风险较高的场景

以下场景如果被重放，可能产生明显业务影响：

1. **非幂等写操作**
    - 创建资源
    - 发起审批
    - 新增授权关系
    - 发放优惠、积分、额度
    - 触发一次性任务

2. **状态推进类操作**
    - 从“待处理”变成“已处理”
    - 从“未授权”变成“已授权”
    - 从“未启用”变成“已启用”

3. **具备副作用的内部回调**
    - 重发消息
    - 重复刷新权限
    - 重复同步配置
    - 重复写审计记录

4. **涉及资金、配额、资格、发券的接口**
    - 这类接口一般不应仅依赖时间戳签名
    - 必须具备额外的幂等或一次性请求保护

---

## 结合内网部署，风险应如何判断

在“纯内网部署”前提下，这个改动通常不是高危漏洞，但也不能简单视为“绝对安全”。

### 可以认为风险相对可接受的条件

如果同时满足以下条件，当前方案通常是可接受的：

1. 服务间通信走受控内网
2. 服务间启用了 TLS 或至少链路可信
3. 请求原文、签名、关键头不会被明文打印到日志
4. 内部服务节点权限管控较严格
5. 主要调用以查询、校验、同步类接口为主
6. 高价值写操作已在业务层自行做幂等保护

### 仍然不能掉以轻心的原因

即使在内网，也仍然存在以下现实风险：

1. **内部节点被攻陷**
    - 一台机器失陷后，攻击者往往最容易得到的不是密钥，而是现成的合法请求

2. **调试日志比链路窃听更常见**
    - 实际项目里，最容易泄露请求材料的往往不是网络抓包，而是开发/测试/运维日志

3. **微服务调用链较长**
    - 请求经过的节点越多，被记录、转储、采样、缓存的机会越多

4. **“内网可信”不是绝对边界**
    - 容器平台、共享中间件、跳板机、代理服务都可能扩大攻击面

因此，更准确的判断应该是：

- **在内网场景下，去掉 nonce 是可接受的工程权衡**
- 但它意味着系统接受了“短时间窗口内可被重放”的残余风险
- 这个风险是否可接受，取决于接口的业务价值和幂等性设计

---

## 当前方案最核心的安全结论

### 结论 1：它不是伪造漏洞

去掉 nonce 并不意味着攻击者可以随便构造请求。

只要密钥没有泄露：

- 攻击者仍然不能伪造任意合法签名
- 攻击者仍然不能篡改报文后继续通过校验

### 结论 2：它是重放风险上升

去掉 nonce 的本质影响是：

- 合法请求的“一次性”保证消失了
- 请求变成“在时间窗口内可重复使用”

### 结论 3：影响大小取决于业务接口是否幂等

- 如果接口本身幂等，风险通常较低
- 如果接口是非幂等写操作，风险会明显上升

---

## 建议的缓解策略

如果决定继续保留“无 nonce”的轻量方案，建议至少补以下防线。

### 1. 对高价值写接口单独做幂等保护

适用场景：

- 创建单据
- 发券
- 扣减额度
- 变更关键状态
- 外部回调

建议方式：

- 业务请求号
- 幂等键
- 去重表
- 状态机防重复推进

说明：

- 幂等保护不一定必须放在安全签名层
- 放在业务层通常更贴近真实语义

### 2. 缩小时间窗口

如果当前 `allowedSkewSeconds` 较大，可以适当收紧。

收益：

- 减少可重放时间窗

代价：

- 对服务间时钟同步要求更高
- 时钟漂移更容易导致误拒绝

### 3. 严控日志输出

必须避免完整打印以下内容：

- signature
- timestamp
- callerService
- gateway 用户上下文 payload
- 原始内部请求头

因为很多重放并不是来自抓包，而是来自日志复制。

### 4. 服务间尽量启用 TLS

即使在内网，也建议尽量使用可信链路，减少报文被旁路观察的机会。

### 5. 对特别敏感的内部接口恢复一次性请求机制

不是所有接口都必须重新引入 nonce。

可以采用分级策略：

- 普通查询/普通内部调用：保留当前 `timestamp + signature`
- 高价值写操作/一次性动作：恢复 `nonce` 或业务幂等键

这通常比“所有接口都强制 nonce”更平衡。

---

## 推荐的风险分级结论

### 低风险

- 查询类接口
- 幂等更新接口
- 重复执行无明显副作用的内部同步调用

### 中风险

- 会刷新缓存、重复下发事件、重复写普通操作记录的接口
- 虽然不会直接造成资金/权限问题，但可能引起状态抖动或重复执行

### 高风险

- 资金、配额、发券、授权、审批推进、资源创建等非幂等接口
- 这类接口不建议仅依赖 `timestamp + signature`

---

## 是否需要恢复 nonce

建议不要用“一刀切”的方式判断，而应按接口类别决定。

### 可以不恢复 nonce 的情况

- 纯内网
- 链路可信
- 请求材料不落日志
- 接口大多是查询或天然幂等
- 关键写操作已有业务幂等

### 建议恢复 nonce 或等价机制的情况

- 存在高价值非幂等操作
- 链路中有多层代理、日志、APM 采样
- 某些接口一旦重复执行会带来明显损失
- 需要满足更严格的审计或合规要求

---

## 建议给项目维护者的最终结论

可以将当前方案定义为：

> 在内网微服务场景下，为降低 Redis 对认证链路的耦合，签名协议简化为 `timestamp + signature`。该方案可以继续提供请求来源校验、完整性校验和有限时间窗口约束，但不再提供严格的防重放能力。因此，对于查询类和幂等类接口，该风险通常可接受；对于高价值、非幂等写操作，应额外引入业务幂等、请求去重或恢复一次性请求机制。

---

## 后续建议

建议下一步在项目内补一份接口分级清单：

1. 哪些接口属于查询/幂等接口，可继续使用当前方案
2. 哪些接口属于高价值写操作，必须增加幂等键或去重机制
3. 哪些日志需要脱敏或禁止打印完整签名材料
4. `allowedSkewSeconds` 是否需要根据生产环境时钟情况进一步收紧

这样可以把“去掉 nonce”从一个笼统判断，变成一个可治理、可审计、可落地的安全决策。
